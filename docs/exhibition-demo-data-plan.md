## 1. 문제 분석

### 1-1. Analytics 데이터 소실 원인

현재 시드 스크립트의 데이터 생성 방식이 **실행 시점 기준 rolling 24h window**로 되어 있어, 시드 실행 후 24시간이 지나면 조회 범위 밖으로 밀려남.

| 스크립트 | 대상 장치 | 시간 범위 | 간격 | 포인트 수 |
|----------|-----------|-----------|------|-----------|
| `seed.ts` | 전체 Installation (~40개) | 실행 시점 기준 과거 24h | 1시간 | 24개/장치 |
| `seedTelemetry.ts` | **3개만** (SONGDO01, SONGDO02, RNDTEST5) | 실행 시점 기준 과거 24h | 30분 | 48개/장치 |

**핵심 문제:**
1. 시드 실행 후 24시간이 지나면 `GET /devices/:id/readings?hours=24` 쿼리 결과가 빈 배열
2. `seedTelemetry.ts`는 3개 장치만 대상 → 나머지 37개 장치는 `seed.ts`의 1시간 간격 데이터만 존재
3. 전시회 기간 (5/4~5/10) 전체를 커버하는 데이터가 없음

### 1-2. API 조회 제한

```typescript
// deviceService.ts:654
const clampedHours = Math.min(Math.max(hours, 1), 168); // 최대 7일
```

- 백엔드: 최대 **168시간 (7일)** 까지 조회 가능 → 전시회 7일 커버 가능
- 프론트엔드: `HISTORY_HOURS = 24`로 고정 → 조정 필요할 수 있음

### 2-2. 보상 전/후 데이터값 현실화

고객이 요청한 "현실적인" 보상 효과 시연을 위해, 보상 전후 차이를 극대화.

#### 역률 (Power Factor)

| 필드 | 의미 | 현재 값 범위 | 변경 목표 | 비고 |
|------|------|-------------|-----------|------|
| `tpf1` | **보상 전** TPF | 72~91% | **70~80%** | 비선형 부하 환경의 전형적 역률 |
| `tpf2` | **보상 후** TPF | 86~99% | **96~99%** | PSVG 보상 장치 적용 후 |
| `dpf1` | **보상 전** DPF | 73~92% | **72~82%** | 기본파 역률 |
| `dpf2` | **보상 후** DPF | 87~99% | **97~99%** | 보상 후 기본파 |

**전시 효과:** 보상 전 ~75% → 보상 후 ~98% (약 **23%p 개선**)

#### 전류 고조파 왜곡률 (THD-i)

| 필드 | 의미 | 현재 값 범위 | 변경 목표 | 비고 |
|------|------|-------------|-----------|------|
| `loadCurrentTHDL1~L3` | 부하측 THD-i (**보상 전**) | 2~12% | **20~35%** | VFD/인버터 부하 환경 |
| `gridCurrentTHDL1~L3` | 계통측 THD-i (**보상 후**) | 1~9% | **1.5~3.5%** | 능동형 고조파 필터 적용 후 |

**전시 효과:** 보상 전 THD-i ~28% → 보상 후 ~2.5% (약 **90% 이상 저감**)

> **현실성 근거:** THD-i 20~35%는 대형 아파트 단지 지하주차장(EV 충전기 밀집), 데이터센터(UPS/서버 전원), 산업 현장(VFD 구동 모터) 등에서 실제 관측되는 수치. IEEE 519-2022 기준 일반 수용가 THD-i 한도 8% 초과 상태.

#### 전력 (S/P/Q/H) 보상 전후 차이

| 항목 | 의미 | 조정 방향 |
|------|------|-----------|
| `uncompS` vs `compS` | 피상전력 (kVA) | 보상 후 **15~25% 감소** |
| `uncompQ` vs `compQ` | 무효전력 (kvar) | 보상 후 **70~85% 감소** (핵심 지표) |
| `uncompH` vs `compH` | 고조파 전력 | 보상 후 **85~95% 감소** |
| `uncompP` vs `compP` | 유효전력 (kW) | 거의 변동 없음 (±2%) |

### 2-3. Fault 이벤트 시드 추가

전시회 기간 중 **간헐적 fault** 데이터를 `FaultEvent` + `ModuleFaultState` 테이블에 삽입.

#### FaultEvent 시드 계획

| 항목 | 내용 |
|------|------|
| 대상 장치 | 전체 중 **5~8개** (각 클라이언트별 1~2개) |
| 빈도 | 장치당 **2~5건** / 7일 (총 약 20~30건) |
| 시간 분포 | 전시 기간 전체에 걸쳐 불규칙 분포 |
| 해결 상태 | 대부분 CLEAR (1~2건만 미해결 유지) |

#### 현실적인 Fault 유형 목록

| desc (최대 48자) | 심각도 | 비고 |
|------------------|--------|------|
| `Over Temperature` | 중 | 가장 흔한 고장 — 여름철 빈번 |
| `Fan Failure` | 중 | 냉각팬 고장 |
| `IGBT Short Circuit` | 상 | IGBT 단락 (즉시 모듈 정지) |
| `DC Link Over Voltage` | 상 | DC 링크 과전압 |
| `Output Over Current` | 중 | 출력 과전류 |
| `Communication Timeout` | 하 | Modbus 통신 타임아웃 |
| `Capacitor Degradation` | 중 | 커패시터 열화 감지 |
| `Input Fuse Blown` | 상 | 입력 퓨즈 단선 |

#### ModuleFaultState 동기화

- fault 발생 장치의 `moduleStatus[]` 내 해당 모듈 인덱스를 `3` (FAULT)으로 설정
- 대부분은 `resolvedAt` 값 존재 (CLEAR 처리됨)
- **1~2건만 `resolvedAt = null`** → 대시보드에 active fault로 표시

## 4. 시드 데이터 생성 로직 설계

### 4-1. 시간 패턴 (24시간 주기)

```
00:00~06:00  야간 — 부하 60%, 온도 낮음, fault 발생 희소
06:00~09:00  기상 — 부하 상승 (70→100%)
09:00~18:00  주간 — 부하 100%, 피크 13:00~15:00 (110%)
18:00~22:00  저녁 — 부하 90→80%
22:00~24:00  심야 — 부하 70→60%
```

### 4-2. 요일 패턴

- **평일** (월~금): 기본 부하 100%
- **주말** (토~일): 기본 부하 70~80%
- 데이터센터는 주말 차이 없음 (24/7 운영)

### 4-3. 장치별 변동

- 각 장치마다 고유 위상(phase) offset → 동일 데이터 방지
- 소음(noise) ±2~5% 랜덤 추가 → 자연스러운 데이터
- 사이트 유형별 프로필:
  - **데이터센터**: 고부하(80~95A), 안정적 변동, THD 낮음(20~25%)
  - **아파트 단지**: 중부하(30~55A), 주야간 변동 큼, THD 중간(25~30%)
  - **지하주차장**: 저~중부하(25~35A), 간헐적 EV 충전 피크, THD 높음(28~35%)
  - **R&D 연구소**: 저부하, 불규칙 패턴

## 6. 주의사항

1. **기존 운영 데이터 보존**: 전시회 기간 이외의 기존 TelemetryRecord는 건드리지 않음
2. **타임존**: `recordedAt`은 UTC 저장 → KST(+9) 고려하여 `2026-05-03T15:00:00Z` ~ `2026-05-11T14:59:59Z` 범위 사용
3. **멱등성**: 재실행 시 해당 기간 데이터 삭제 후 재삽입
4. **프론트엔드 대응**: 전시회 기간 중 차트가 7일 데이터를 표시하려면 `HISTORY_HOURS` 변경 또는 날짜 선택 UI 추가 검토 필요
5. **Fault 데이터**: `requireAdmin` 미들웨어로 보호되므로 전시 시연 시 admin 계정으로 접속 필요
